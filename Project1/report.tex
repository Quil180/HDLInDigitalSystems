\section*{Introduction}
% ============================================================

This report documents the High-Level Synthesis (HLS) of a Sobel Edge Detection filter using Vivado HLS 2019.1.
The design processes a 256$\times$256 grayscale image and produces an edge-detected output using the Sobel operator.
The filter applies 3$\times$3 horizontal ($G_x$) and vertical ($G_y$) convolution kernels and computes the gradient
magnitude $|\nabla I| \approx |G_x| + |G_y|$, clamped to [0,\,255].

The design was synthesized with default settings (no HLS pragmas) targeting a Zynq-7020
(\texttt{xc7z020clg484-1}) with a 10\,ns clock period.

\section{Question 1: Synthesis Results}
% ============================================================

\subsection{1a) Default Synthesis Metrics}

The design was synthesized using default Vivado HLS scheduling with no optimization pragmas applied.
Table~\ref{tab:synth} summarizes the key metrics from the C-synthesis report.

\midrule

\end{table}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{screenshots/summary.png}
  \caption{Vivado HLS Synthesis Summary for \texttt{sobel\_filter}, showing timing, latency, and utilization estimates.}
  \label{fig:synth_summary}
\end{figure}

The estimated clock period of 8.253\,ns is well below the 10\,ns target, confirming the design meets timing with
1.747\,ns of positive slack. The design is not pipelined (0 out of 4 loops pipelined), which explains the high
worst-case latency of 2,359,809 cycles---each of the $256 \times 256 = 65{,}536$ inner pixels must be processed
sequentially through the 3$\times$3 kernel convolution.

The latency variance between best-case (131,585) and worst-case (2,359,809) arises from the conditional
border-pixel check: border pixels skip the kernel convolution entirely (2 cycles), while interior pixels
execute the full nested kernel loop (up to 36 cycles per pixel).

The two DSP48E blocks are bound to multiply-accumulate (MAC) operations
(\texttt{i0 + i1 * i2}) within the kernel loop, where each source pixel is multiplied by its
corresponding Sobel coefficient. The 295 FFs absorb the FSM state encoding, loop counters
(\texttt{x\_0\_reg}, \texttt{y\_0\_reg}, \texttt{kx\_0\_reg}, \texttt{ky\_0\_reg}), partial sum
accumulators (\texttt{sum\_x}, \texttt{sum\_y}), and pipeline registers. The 687 LUTs cover
address generation, comparison logic, absolute-value select muxes, and the magnitude addition.

\subsection{1b) Analysis Perspective}

The Analysis Perspective provides a cycle-level schedule view of how operations are mapped to FSM states.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\textwidth]{screenshots/analysis.png}
  \caption{Analysis Perspective showing the schedule of operations across FSM states in \texttt{sobel\_filter}.}
  \label{fig:analysis}
\end{figure}

The schedule viewer reveals how each iteration of the innermost kernel loop requires multiple FSM states for:
memory reads from the \texttt{src} array (via the \texttt{ap\_memory} interface), multiply-accumulate
operations using the Sobel coefficients accessed from the \texttt{Gx} and \texttt{Gy} distributed ROMs,
and the final gradient magnitude computation with clamped write to \texttt{dst}.

The innermost loop (\texttt{Loop 1.1.1.1}) has a latency of 3 cycles per iteration with a trip count of 3,
giving 9 cycles per kernel-row pass. The kernel-column loop (\texttt{Loop 1.1.1}) wraps this with latency
11 cycles $\times$ 3 iterations = 33 cycles for the full 3$\times$3 convolution per pixel.

\subsection{1c) Resource Profile}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\textwidth]{screenshots/resource.png}
  \caption{Resource Profile showing DSP48E allocation and memory bindings within \texttt{sobel\_filter}.}
  \label{fig:resource}
\end{figure}

The Bind Op Report confirms both DSP48E blocks are allocated to the \texttt{sobel\_filter} module
for the MAC operations (\texttt{sobel\_filter\_mac\_muladd\_8ns\_3s\_32ns\_32\_1\_1}), each computing
\texttt{pixel * coefficient + accumulator} within the inner product loop.

The Bind Storage Report shows the two Sobel kernels implemented as small distributed ROMs:
\begin{itemize}
  \item \texttt{Gx\_U} (sobel\_filter\_Gx): 9 words $\times$ 3 bits, using 3 FFs + 1 LUT
  \item \texttt{Gy\_U} (sobel\_filter\_Gy): 9 words $\times$ 3 bits, using 3 FFs + 1 LUT
\end{itemize}
No BRAM or URAM is used. The input/output arrays are accessed through external \texttt{ap\_memory}
interfaces with 16-bit addresses (sufficient for $2^{16} = 65{,}536$ elements), 8-bit data for
\texttt{src\_q0} (read), and 8-bit data for \texttt{dst\_d0} (write).

\section{Question 2: C/RTL Co-simulation}
% ============================================================

\subsection{2a) Co-simulation Report}

C-simulation was first run to verify functional correctness. The testbench generates a synthetic
256$\times$256 image with vertical stripe patterns (period 32 pixels, alternating between intensity
200 and 50) and a horizontal bar (intensity 255), applies the Sobel filter, and compares every output
pixel against a software reference implementation.

The C-simulation output confirms zero mismatches:
\begin{verbatim}
Non-zero output pixels: 4264 / 65536
TEST PASSED (4264 edge pixels detected)
\end{verbatim}

C/RTL co-simulation was then run using the synthesized Verilog model under XSIM.
The same testbench stimulus is fed into the RTL, and outputs are compared against the C model.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{screenshots/cosim.png}
  \caption{C/RTL co-simulation report showing PASS status with measured latency and interval.}
  \label{fig:cosim}
\end{figure}

The co-simulation completed with status \textbf{PASS}, confirming the synthesized RTL is
functionally equivalent to the C model. The simulation log reports:

\begin{verbatim}
RTL Simulation : 1 / 1 [100.00%] @ "23251435000"
$finish called at time : 23251475 ns
\end{verbatim}

The measured RTL latency of approximately 23.25\,ms (at the 10\,ns clock period) corresponds to
$\sim$2,325,143 cycles, consistent with the synthesis estimate of 2,359,809 worst-case cycles.
The small difference arises because the actual test image has some border pixels that take fewer
cycles than the worst case.

Since only one transaction was simulated, the initiation interval (II) is reported as N/A.
For this non-pipelined design, the next input can be accepted after latency~$+$~1 clock cycles.

\subsection{2b) Waveform Trace Analysis}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\textwidth]{screenshots/waveform.png}
  \caption{Co-simulation waveform showing the \texttt{ap\_ctrl\_hs} handshake signals and memory interface activity.}
  \label{fig:waveform}
\end{figure}

The waveform trace confirms the standard \texttt{ap\_ctrl\_hs} block-level protocol:
\begin{itemize}
  \item \texttt{ap\_start} is asserted by the testbench to initiate image processing.
  \item The design enters the active computation phase, sequentially reading pixels from \texttt{src}
        and writing edge-detected results to \texttt{dst}.
  \item \texttt{ap\_done} pulses high once all 65,536 output pixels have been written.
  \item \texttt{ap\_ready} signals that the design can accept a new input transaction.
  \item The \texttt{src\_address0} and \texttt{dst\_address0} signals show the sequential
        memory access pattern, sweeping through the image row by row.
\end{itemize}

The total active computation window visible in the trace spans the full $\sim$23.25\,ms latency,
dominated by the un-pipelined sequential traversal of all image pixels and their 3$\times$3 neighborhoods.

\subsection{2c) Export RTL}

The C-synthesis step generates both VHDL and Verilog RTL under the solution directory:
\begin{itemize}
  \item \texttt{sobel\_hls/solution1/syn/verilog/} --- Verilog sources
  \item \texttt{sobel\_hls/solution1/syn/vhdl/} --- VHDL sources
\end{itemize}

\noindent The generated RTL consists of:
\begin{itemize}
  \item \textbf{sobel\_filter.v}: Top-level module containing the 8-state FSM
        (\texttt{ap\_CS\_fsm}), loop counters (\texttt{y\_0\_reg}, \texttt{x\_0\_reg},
        \texttt{ky\_0\_reg}, \texttt{kx\_0\_reg}), partial-sum accumulators, address generation
        logic, multiplexers, and instantiations of the DSP-based MAC and ROM primitives.

  \item \textbf{sobel\_filter\_Gx.v / sobel\_filter\_Gy.v}: Distributed ROM modules storing
        the Sobel kernel coefficients as 9-entry $\times$ 3-bit lookup tables. Implemented using
        FFs and LUTs rather than BRAM, appropriate for such small constant arrays.

  \item \textbf{sobel\_filter\_mac\_muladd\_8ns\_3s\_32ns\_32\_1\_1.v}: The multiply-accumulate
        primitive (\texttt{i0 + i1 * i2}) mapped to DSP48E slices. Two instances are used---one
        for the $G_x$ accumulation and one for the $G_y$ accumulation.
\end{itemize}

\textbf{Note:} The IP Catalog packaging step (\texttt{export\_design -format ip\_catalog}) fails
on Vivado HLS 2019.1 due to a known timestamp overflow bug where \texttt{core\_revision} (a Unix
timestamp) exceeds the 32-bit signed integer range after year 2024. This is a tool bug, not a
design issue. The synthesized RTL is fully available and functionally verified in the solution
directory.

\section*{Closing Notes}
% ============================================================

The default synthesis run produces a working Sobel Edge Detection filter that correctly processes
a 256$\times$256 grayscale image on the Zynq-7020 target. The design achieves an estimated clock
frequency of $1/8.253\,\mathrm{ns} \approx 121\,\mathrm{MHz}$, uses 2 DSP48E blocks, 295 FFs,
and 687 LUTs with zero BRAM---representing less than 2\% utilization of the target device.

Without any optimization pragmas, the four nested loops execute sequentially, giving a worst-case
latency of 2,359,809 cycles ($\sim$23.6\,ms per frame, or $\sim$42 frames/second).
Part~2 of this project will explore the design space by applying HLS directives (loop pipelining,
array partitioning, loop unrolling, dataflow) to reduce latency and improve throughput at the
cost of additional resource usage.
